# MCP Snippet Inlining

## What Gets Inlined

When `augx export` writes an aggregated file, it may inline a single tool-agnostic snippet describing how the consuming AI agent should query the **Beads MCP server** for live task state. The snippet body comes from `augment-extensions/workflows/mcp/templates/beads.md` and contains, at minimum:

- The server id (`beads`).
- The relevant public operations (`tasks/list`, `tasks/get`, `dependencies/list`, `status/get`).
- A reference to `augment-extensions/workflows/beads/` for the human-readable command guide.
- A directive that the consuming agent MUST query the MCP server for task context and NEVER rely on prose in the exported file.

The snippet contains **no** task IDs, statuses, dependency lists, endpoint URLs, secrets, or other live data. It is pure static guidance.

## Trigger Condition

The snippet is inlined **only when** at least one of the following declares an MCP server with id `beads`:

- `.vscode/mcp.json`
- `.augment/mcp/servers.json`

When neither file exists, or neither declares the `beads` server, the snippet is omitted entirely. The exported file remains focused on the static module guidance and does not reference MCP at all.

This keeps the trigger explicit and auditable: presence of an MCP config that names `beads` is the sole signal.

## Inlining Position

The snippet appears immediately after the generated banner (and, for Cursor, after the YAML frontmatter), before the first `## Module:` section header. Per-target adapters MAY wrap it in a single tool-specific line (e.g., a Cursor `> Note:` callout) but MUST NOT alter the substantive content.

## What This Looks Like In Practice

`CLAUDE.md` (with `beads` declared in `.vscode/mcp.json`):

```
<!-- GENERATED BY augx export - DO NOT EDIT -->
<!-- augx-version: 1.0.0 -->
<!-- generated-at: 2026-05-25T12:00:00.000Z -->
<!-- source-hash: a1b2c3d4e5f6a7b8 -->
<!-- modules: coding-standards/typescript@1.0.0, workflows/openspec@1.0.0 -->

### Live Task Context via Beads MCP
...

## Module: coding-standards/typescript (v1.0.0)
...
```

`CLAUDE.md` (without `beads` declared anywhere):

```
<!-- GENERATED BY augx export - DO NOT EDIT -->
<!-- augx-version: 1.0.0 -->
<!-- generated-at: 2026-05-25T12:00:00.000Z -->
<!-- source-hash: a1b2c3d4e5f6a7b8 -->
<!-- modules: coding-standards/typescript@1.0.0, workflows/openspec@1.0.0 -->

## Module: coding-standards/typescript (v1.0.0)
...
```

The body bytes differ; the source hash will differ; the modules list is unchanged. Drift detection treats this correctly: removing the Beads MCP declaration and re-running `augx export` is a normal write, not a drift refusal.

## Why MCP-First

The cross-platform feature is explicitly **MCP-first** for task context: exported files never embed live task data because:

1. **Staleness**: a `bd-` ID baked into `CLAUDE.md` at export time may already be closed by the time an agent reads it.
2. **Surface area**: spreading task data across four target files multiplies the surface where stale data can live.
3. **Auditability**: a single MCP query point is far easier to audit than four divergent prose copies.
4. **Privacy**: task descriptions sometimes contain sensitive references; the MCP server can scope responses, exported files cannot.

The static snippet is the agreed-upon "pointer", not the data itself.

## When To Remove The Snippet

The snippet disappears from exported files on the next `augx export` run after either:

- The `beads` MCP server declaration is removed from both `.vscode/mcp.json` and `.augment/mcp/servers.json`.
- A future `--ignore-mcp` flag is passed to `augx export` (reserved for a later change, not in scope today).

## Template Source

`augment-extensions/workflows/mcp/templates/beads.md` is the canonical template. It is versioned together with the `cross-platform` capability. Changes to the template (e.g., adding a new MCP operation like `comments/append`) cause the source hash to recompute on the next export run, which produces a normal write on every target.

## What If A Project Declares A Different MCP Server?

Only the `beads` server triggers the snippet. A project may declare other MCP servers in `.vscode/mcp.json` or `.augment/mcp/servers.json` without affecting cross-platform export behavior. The trigger logic is a literal lookup for an entry whose key (or `id`) is `beads`; no fuzzy matching, no other server names recognized in this change.

Future capabilities may add per-server snippets (e.g., a `filesystem` MCP server template). Each would be opt-in and conditioned on its own declaration.

## Validation

The template MUST satisfy a content check at build time (see `tests/mcp-template.test.ts`):

- Contains the string `Beads MCP server`.
- Contains all four operation names: `tasks/list`, `tasks/get`, `dependencies/list`, `status/get`.
- Contains NO `bd-` issue IDs.
- Contains NO status keywords used as data assertions.

This guarantees the snippet stays pure-guidance and never leaks data.
